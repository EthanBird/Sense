# ADR 0008：M6 输入精修、语义候选与连续面板

- 状态：Accepted
- 日期：2026-07-23
- 阶段：M6 输入精修
- 版本：`0.3.3-m6`

## 背景

`0.3.2-m5` 已经加入中文渐进分词、20,000 词英文资产与可滚动 Emoji 面板，但真实输入继续暴露出若干会直接破坏日常使用的问题：

1. 最后一个 composing 字母被退格删除时，调用 `finishComposingText()` 会把宿主中的 composing span 固化，形成残留英文字母；
2. 无输入时的“中 · 先思输入法”品牌行占用高度，工具栏又使用独立颜色，键盘被切成不必要的两层；
3. 中文模式上滑 `S/L` 仍输出半角 `!/?`，英文模式则没有独立的词典组合态；
4. Emoji 数量和滚动手感不足，符号页既没有分类，也无法覆盖数学、音标和多语种字符；
5. Emoji 与常用符号只能从面板寻找，不能由 `ji`、`oumu` 等拼音进入候选；
6. 折叠候选只显示固定前缀，用户必须展开后才能访问后续候选，且缺少可预测的拖动、fling 与对齐行为。

这些问题横跨 `InputConnection`、候选协议、触控状态和数据目录。只在 Canvas 上调整坐标无法形成可回归的质量边界。

## 决策

### 1. 空 composition 必须替换宿主 composing span

输入法在 composition 非空和变空时都调用同一条“设置 composing 文本”路径。空状态使用：

```text
setComposingText("", 1)
```

而不是只调用 `finishComposingText()`。这样退格删除最后一个中文拼音或英文组合字母时，宿主中对应的 composing span 被替换为空；只有正常确认上屏时才结束 composing。

该语义由纯 Kotlin 的 `EditorComposingTextUpdate` 固定，中文和英文状态机共用同一回归。`onUpdateSelection()` 还会校验宿主报告的 composing 范围：若光标不再折叠在 composing 末端，或宿主已撤销该范围，输入法先清空本地 session，再结束宿主 span，避免旧 composition 在新光标处被重复建立。

### 2. 英文模式使用独立的 revisioned 组合态

英文按键不再逐字永久提交。`EnglishInputSession` 持有：

- 当前英文 composing 文本；
- 单调递增 revision；
- 由 20,000 词本地英文资产生成的候选快照；
- 只允许当前 revision 提交的候选选择；
- 仅在存在精确词时供空格或分隔符采用的默认候选。

输入 `host` 时首三项固定为 `host, hosts, hostile`。输入尚未形成精确词时，空格和标点不能静默选择更长的补全；原始字母仍可直接确认。大小写只影响显示与上屏值，不改变词典索引。

### 3. 顶部区域在“工具栏”和“候选”之间复用

无 composition 时，工具栏直接占据键盘顶部，不再绘制品牌/模式行。工具栏底色与键盘底板一致，不创建独立色带。

有候选时，候选条复用顶部区域。折叠条同时支持：

- 慢速拖动：像素级跟随手指，松手不强制跳页；
- 快速 fling：最多推进一个可见页宽，防止跳过候选；
- 结束对齐：快速滑动按候选边界吸附；
- 边界阻尼：越界有阻力且最终回到合法范围；
- 展开面板：继续使用全局候选索引与显式前/后页。

滚动成立后取消同一 pointer 的候选点击；多 pointer 的普通字母输入仍保持独立状态。

### 4. 标点显示和值都由输入模式决定

`SwipeCharacterMap` 的键位不变，但 `S/L` 上滑值根据语言模式返回：

- 中文：`！`、`？`
- 英文：`!`、`?`

上滑提示与实际提交必须调用同一个映射，不能出现键帽显示全角而提交半角的分叉。

### 5. Emoji 使用离线目录和连续动力学滚动

Emoji 数据采用稳定、不可变的离线目录，共八类、1,310 项，覆盖表情、人物、自然、食物、活动、出行、物品、符号/旗帜。面板采用连续纵向像素滚动：

- touch slop 前仍可点击；
- 超过阈值后锁定为滚动并取消元素提交；
- 松手后按有界速度和摩擦继续；
- 到达边界后停止，不产生页面跳变。

目录和滚动物理均为纯 Kotlin 可测试数据；Android View 只负责把 MotionEvent 时间/位移输入状态机并逐帧绘制。

### 6. 符号页采用分类轨道与可滚动网格

符号页拆为固定分类轨道与当前分类字符网格，共 22 类、1,947 项：

`常用、中文、英文、网络、数学、角标、序号、音标、平假、片假、箭头、特殊、拼音、注音、竖标、部首、俄文、希腊、拉丁、制表、土音、藏文`。

类别顺序和字符顺序均由离线 `SymbolCatalog` 冻结。分类轨道和字符网格可以独立滚动；选择新分类会把字符网格复位到顶部。长字符串（例如 `.com`、`https://`）与单 code point 符号使用同一提交协议。

### 7. Emoji/符号候选是低优先级的确定性旁路

`SemanticCandidateCatalog` 对规范化后的完整拼音执行精确查找，不做模糊语义推断。首批阻断映射为：

| 拼音 | 候选 |
|---|---|
| `ji` | `🐔` |
| `yao` | `💊` |
| `you` | `🈶`、`→` |
| `suo` | `🔒` |
| `oumu` | `Ω` |
| `pai` | `π` |
| `zuo` | `←` |

语义候选默认插入索引为 6，即位于第一行靠后位置；不得改变前六个主中文候选的相对顺序。文本去重、总量上限和弱 `matchKind` 仍由纯 Kotlin mixer 强制执行。它不进入用户词学习，也不替代中文词典和 Bigram。

### 8. 版本仍停留在 0.3.x

本轮固定为 `versionCode 9`、`versionName 0.3.3-m6`、tag `v0.3.3-m6`，继续以 prerelease 发布。

这里的 M6 表示真实输入反馈驱动的第六轮工程门禁。原路线图中的事件底座顺延到 M7；本轮不宣称事件归档、AI Harness 或真机质量已经完成。

## 发布门禁

发布必须同时验证：

- 退格删除最后一个中文/英文 composing 字符时产生空替换，不在宿主留下字母；
- 英文模式 `host → host, hosts, hostile`、大小写映射和 stale revision 拒绝；
- 规定的 Emoji/符号拼音映射、`you` 双候选顺序、索引 6 插入和主候选顺序保持；
- 中文/英文上滑标点分别为全角/半角；
- 折叠候选慢速跟手、快速最多一页、边界阻尼和候选边界吸附；
- Emoji touch slop、滚动锁定、fling 有界减速以及八类/1,310 项目录基线；
- 22 类符号均非空，数学、角标、音标、平假、片假、箭头、希腊、制表和藏文代表值可达；
- M6 英文组合 p95 不超过 1 ms，语义查找与 mixer p95 各不超过 250 µs；
- M0–M5 既有 Host 门禁、三项模型资产哈希、APK 许可副本与无 `INTERNET` 权限继续通过；
- APK 元数据为 code 9 / `0.3.3-m6`，Debug、Benchmark、Macrobenchmark APK、Lint 和单元测试全部成功。

## 结果与代价

- 空 composing 更新修复宿主残留，但不同 OEM 对 `InputConnection` 的实现仍需真机覆盖，尤其是浏览器、WebView 和富文本编辑器。
- 英文候选不再是中文模式的附属结果，行为更可预测；代价是语言切换、空格、回车和标点都必须显式处理两个 session。
- 折叠候选和面板滚动更接近原生列表，但 Canvas View 需要维护 pointer 锁定、速度与动画失效；Host 物理测试不能替代 60/90/120 Hz 帧时间。
- Emoji 与符号目录显著扩大可达字符范围，同时增加源码和字体兼容差异；缺字时由系统字体决定回退表现。
- 精确拼音语义目录体积小、排序确定，但覆盖有限。后续扩展必须有明确用例和回归，不能变成会压过语言候选的开放语义搜索。
